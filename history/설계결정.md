# 설계 결정 — 절충안 모음

이 프로젝트에서 "일반적인 방법" 대신 의도적으로 단순한 방향을 선택한 부분들.  
규모가 작고 개인 사용 목적이므로 복잡도를 낮추는 쪽을 우선했다.

---

## 1. symbol 문자열 연결 (FK 없음)

**해당 필드**: `Trade.symbol`, `Memo.symbol` → `StockMaster.symbol`

**일반적인 방법**
```sql
Trade.stockMasterId  →  StockMaster.id  (FK, 참조 무결성 보장)
```

**지금 방식**
```sql
Trade.symbol = "삼성전자"  -- 그냥 문자열, FK 없음
```

**선택 이유**  
StockMaster에 등록되지 않은 종목도 거래 입력이 가능해야 함.  
FK였으면 마스터 없는 종목은 거래 입력 불가 → 사용 흐름이 부자연스러움.

**단점 및 보완**  
- 종목명 오타 시 연결이 조용히 깨짐
- 종목명 변경 시 연결된 Trade들을 수동으로 일괄 업데이트해야 함
- 보완: 마스터 등록/수정 시 같은 symbol의 Trade들을 코드로 일괄 업데이트하는 로직 추가

**개선 시점**  
종목 수가 많아지거나, 종목명 변경이 잦아지면 FK 도입 고려.

---

## 2. 태그를 쉼표 구분 문자열로 저장

**해당 필드**: `StockMaster.tags`

**일반적인 방법**
```sql
Tag:              id, name
StockMasterTag:   stockMasterId, tagId  (중간 테이블)
```

**지금 방식**
```sql
StockMaster.tags = "반도체,대형주,배당"  -- 쉼표 구분 문자열
```

**선택 이유**  
SQLite는 배열 타입이 없어 인덱스 기반 정확한 검색이 어려움.  
태그 수가 적고 필터링을 클라이언트(JS)에서 `split(',').filter(Boolean)`으로 처리하므로 충분함.  
별도 테이블 없이 마이그레이션 부담도 없음.

**단점**  
- DB 레벨에서 LIKE 검색 시 `"반도체"`가 `"반도체장비"`도 걸릴 수 있음 (클라이언트 필터링으로 우회)
- 태그 이름 변경 시 전체 레코드 문자열 치환 필요
- 태그별 사용 종목 수 집계가 번거로움

**개선 시점**  
태그를 여러 엔티티에서 공유하거나, 태그 자체를 관리(이름 변경·통계)해야 할 때 정규화.  
PostgreSQL 전환 시 배열 타입(`TEXT[]`) 또는 JSON 사용 가능.

---

## 3. 날짜를 String으로 저장

**해당 필드**: 모든 테이블의 `createdAt`, `updatedAt`, `date`

**일반적인 방법**
```sql
createdAt  DATETIME  또는  TIMESTAMP
```

**지금 방식**
```sql
createdAt  String  -- "2024-01-15T09:30:00.000Z" (ISO 8601)
```

**선택 이유**  
SQLite는 네이티브 DateTime 타입이 없고, Prisma가 SQLite용 DateTime을 내부적으로 문자열로 저장함.  
처음부터 String으로 명시해 타입 불일치 혼란을 방지.  
ISO 8601 문자열은 사전순 정렬이 시간순과 일치하므로 `ORDER BY createdAt` 그대로 동작.

**단점**  
- DB 레벨 날짜 범위 쿼리(`BETWEEN`, `DATE()` 함수)가 어색함
- 타임존 처리가 애플리케이션 레벨에 의존

**개선 시점**  
PostgreSQL 전환 시 `TIMESTAMPTZ`로 자연스럽게 마이그레이션 가능.

---

## 4. 계산 필드를 DB에 저장하지 않음

**해당 필드**: `avgBuyPrice`, `profitAmount`, `profitRate`, `holdingDays`, `isCompleted` 등

**일반적인 방법 (성능 중시)**
```sql
Trade에 avgBuyPrice, profitAmount 컬럼 저장
→ 매매 발생 시 업데이트 트리거 또는 애플리케이션 레벨 업데이트
```

**지금 방식**
```ts
// 조회할 때마다 BuyEntry/SellEntry를 합산해 계산
function enrichTrade(trade): Trade {
  const avgBuyPrice = totalBuyAmount / totalBuyQuantity
  const profitAmount = totalSellAmount - (avgBuyPrice * totalSellQuantity)
  // ...
}
```

**선택 이유**  
데이터 중복 저장 없이 항상 정확한 값 유지.  
BuyEntry/SellEntry를 수정해도 계산 결과가 자동으로 반영됨.  
개인 포트폴리오 규모에서는 성능 문제 없음.

**단점**  
- 종목 수가 수천 개가 되면 목록 조회 시 매번 전체 재계산으로 느려질 수 있음

**개선 시점**  
목록 조회가 체감상 느려지면 `materialized view` 또는 별도 캐시 컬럼 도입.

---

## 공통 원칙

위 네 가지는 모두 같은 판단 기준에서 나왔다.

> **"지금 규모에서 충분히 동작하면, 복잡도를 낮추는 쪽을 선택한다."**

서비스가 커지면 아래 순서로 개선하는 것이 자연스러운 방향이다.

1. symbol FK 도입 (참조 무결성)
2. PostgreSQL 전환 (배열 타입, DateTime, 인덱스)
3. 태그 정규화 (Tag + 중간 테이블)
4. 계산 필드 캐싱 (성능)
